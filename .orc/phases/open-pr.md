You are opening a pull request for completed work in the **horde** project.

Your job: push the branch and create a PR with a clear, descriptive title and body — not just a ticket ID and commit dump.

## Step 1: Gather Context

Determine the work item and branch:
```bash
WORK_ITEM=$(cat "$ARTIFACTS_DIR/current-ticket.txt" 2>/dev/null || echo "$TICKET")
BASE=$(cat "$ARTIFACTS_DIR/base-commit.txt" 2>/dev/null || echo "main")
echo "Work item: $WORK_ITEM, Base: $BASE, Ticket: $TICKET"
```

Read the work item from bd to get the title and description:
```bash
bd show "$TICKET"
```

This gives you the human-readable title (e.g., "DynamoDB store implementation") and the description explaining why this work exists. These are the primary source for the PR title and summary.

In wave mode (`$TICKET` is a wave epic), also look at the children to understand the full scope:
```bash
bd tree "$TICKET" 2>/dev/null || true
```

Read the implementation plan for technical detail:
```bash
cat "$ARTIFACTS_DIR/plan.md" 2>/dev/null || true
```

Get the commit history and change stats:
```bash
git log --oneline $BASE..HEAD
echo "---"
git diff --stat $BASE..HEAD
```

If in wave mode, also read the wave summary:
```bash
cat "$ORC_ARTIFACTS_DIR/wave-summary.md" 2>/dev/null || true
```

## Step 2: Push the Branch

```bash
cd "$PROJECT_ROOT"
BRANCH="horde/${TICKET}"
git push origin "HEAD:refs/heads/${BRANCH}" --force-with-lease 2>&1 || {
    git push origin "HEAD:refs/heads/${BRANCH}" --force 2>&1
}
```

## Step 3: Check for Existing PR

```bash
REPO_SLUG=$(git remote get-url origin | sed -E 's#https?://[^/]+/##; s#git@[^:]+:##; s#\.git$##')
if gh pr view "$BRANCH" --repo "$REPO_SLUG" &>/dev/null; then
    echo "PR already exists for $BRANCH — skipping creation."
fi
```

If a PR already exists, stop here. Do not create a duplicate.

## Step 4: Write the PR

From the bead, plan, and diff, synthesize:

**Title** (under 72 characters):
- Use the bead title from `bd show` — that is the human-readable name for this work
- Strip the sequence number prefix (e.g., "9- DynamoDB store implementation" becomes "DynamoDB store implementation")
- Do not use the ticket ID as the title

**Body** using this structure:

```markdown
## Summary

<2-3 sentences explaining what this PR does and why. Draw from the bead description and plan summary — the bead description has the "why", the plan summary has the "what".>

## Changes

<Bullet list of key changes, organized by package/area. Not a commit-by-commit dump — group related work. Focus on what a reviewer needs to know.>

## Testing

<What tests were added and what they cover>

---

*Ticket: `<TICKET>`*
```

Create the PR:
```bash
gh pr create \
    --repo "$REPO_SLUG" \
    --head "$BRANCH" \
    --base main \
    --title "<title>" \
    --body "$(cat <<'EOF'
<body>
EOF
)"
```

## Rules

- The title describes the change, not the ticket. "DynamoDB store implementation", not "horde-txr".
- The body synthesizes — do not dump raw commits or copy the plan verbatim.
- Keep it concise. A reviewer should understand what changed and why in 30 seconds.
- If neither the bead nor the plan is available, fall back to commit messages and the diff.
- Do not include AI attribution lines (no "Co-Authored-By", "Generated by", etc.).
